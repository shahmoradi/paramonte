13@section ParaMonte What is ParaMonte?
14<!--==============================-->
15
16ParaMonte is a multi-language library of serial and parallel Monte Carlo and Machine Learning routines
17scientific inference, e.g., for sampling mathematical density functions of arbitrary-dimensions,
18with the design goal of unifying
19
20+ the **automation** of simulations and inference,
21+ the **user-friendliness** of the library and routines,
22+ the **accessibility** from multiple programming environments,
23+ the **high-performance** at runtime, and,
24+ the **scalability** across many parallel processors.

102@section ParaMonteLangNamingConventions ParaMonte Python Naming Conventions
103
104+ The **camelCase** naming style is enforced throughout the ParaMonte Python library for all variable and function names.<br>
105+ The **PascalCase** naming style is enforced throughout the ParaMonte Python library for all class names.<br>
106 In other words, **all class names begin with a capital letter**.<br>
107
108[⛓⛓](#ParaMonteLangNamingConventionsVariables)
109@subsection ParaMonteLangNamingConventionsVariables ParaMonte Python Naming Conventions: Variables
110
111By convention in this library,
112
113+ All variables and procedure names except compile-time constants begin with an lowercase letter.
114
115+ All constants and parameters should generally be typed in uppercase.
116
117+ The names of variables that **always** represent vectors of values **may** be suffixed with `Vec` or `Vector`.
118
119+ The names of variables that **always** represent matrices of values **may** be suffixed with `Mat` or `Matrix`.
120
121+ A significant attempt has been made to end all `logical` variables with a passive verb.<br>
122 This is to ensure that the full variable name virtually forms a proposition.<br>
123 In other words, a `logical` variable name should be an English-language statement that evaluates to either `true` or `false`.<br>
124 For example, `parallelismMpiFinalizeEnabled` is one such proposition.<br>
125 + Occasionally, names that begin with the verb `is` can also be used to label `logical` objects.<br>
126 + But as a general rule, names that begin with a verb should be reserved for procedures.<br>
127
128[⛓⛓](#ParaMonteLangNamingConventionsProcedures)
129@subsection ParaMonteLangNamingConventionsProcedures ParaMonte Python Naming Conventions: Procedures
130
131+ Procedure names should be descriptive of the action performed by the procedure. For example,
132
133 + `getCov` means *generate (a) covariance matrix*.
134 + `setMatCopy` means *copy the matrix into the specified buffer*.
135
136+ Procedure names should preferably begin with a lowercase verb.
137
138+ **Exceptions** to the above and below rules are allowed when,
139 + the procedure name is exceptionally famous, or
140 + it is very inconvenient to prefix the procedure name with a verb.
141
142+ The names of functions with return values should preferably begin with <b>`get`</b>.<br>
143 The reasoning is simple: Functions in mathematics *generate and obtain* a **new** object instead of changing (resetting) the state of an existing object.<br>
144 For example, the function `getMatSym(mat)` generates a symmetric version of the input matrix and returns it as the function result.<br>
145 **Exceptions** to this naming convention are allowed, for example, when the `get` prefix is inconvenient or when the function returns a `logical` result.<br>
146
147+ Functions that return objects of type `logical` should be preferably prefixed with `is` or
148 be named such that the name begins with a verb and reads as a proposition, evaluating to
149 either `True` or `False`.

225@section ParaMonteLangDocumentationGuidelines ParaMonte Python Documentation Guidelines
226
227+ **Doxygen custom command orderings**.
228
229 + The Doxygen tag <tt>@@brief</tt> must always be the first line of the documentation of modules, types, and procedures.<br>
230 Example: [pm.sampling.Paradram](@ref Paradram).<br>
231 + The Doxygen tag <tt>@@details</tt>, if it exists, must always immediately follow the Doxygen tag <tt>@@brief</tt>.<br>
232 Example: [pm.sampling.Paradram](@ref Paradram).<br>
233 + The Doxygen tag <tt>@@param</tt>, if any number of it exists, must always immediately follow the Doxygen tag <tt>@@brief</tt> (or <tt>@@details</tt> if it exists).<br>
234 Example: [pm.sampling.Paradram](@ref Paradram).<br>
235 + The Doxygen tag <tt>@@return</tt>, must be exclusively used to indicate the return value of functions.<br>
236 If it exists, it must appear immediately after the set of <tt>@@param</tt> tags. Example: [pm.sampling.Paradram](@ref Paradram).<br>
237 + If a generic interface is being documented, the ParaMonte custom command <tt>@@interface</tt> must appear immediately
238 after the Doxygen <tt>@@return</tt>, <tt>@@param</tt>, <tt>@@details</tt>, or <tt>@@brief</tt> tags in the specified order, if any exists.<br>
239 + The Doxygen tag <tt>@@warning</tt>, if any number of it exists, must immediately follow the Doxygen tag <tt>@@return</tt> if it exists,
240 otherwise <tt>@@param</tt> if it exists, otherwise <tt>@@details</tt> if it exists, otherwise <tt>@@brief</tt>.<br>
241 The <tt>@@warning</tt> tag must be used to highlight situations that require special attention of the user,
242 otherwise, there is a danger for the code section being documented to not behave normally as one may expect.<br>
243 + The Doxygen tag <tt>@@attention</tt> has the same functionality and usage as <tt>@@ </tt>.<br>
244 Therefore, <tt>@@warning</tt> should be preferred wherever <tt>@@attention</tt> is needed.<br>
245 Exceptions are allowed and if they occur, the same documentation conventions as those of <tt>@@warning</tt> also apply to the tag <tt>@@attention</tt>.<br>
246 + The Doxygen tag <tt>@@remark</tt>, if any number of it exists, must immediately follow the Doxygen tag <tt>@@warning</tt> if it exists,
247 otherwise the Doxygen tag <tt>@@return</tt> if it exists, otherwise <tt>@@param</tt> if it exists, otherwise <tt>@@details</tt> if it exists, otherwise <tt>@@brief</tt>.<br>
248 The tag <tt>@@remark</tt> should be reserved for explaining behavior that is **directly** related to the code segment being documented,
249 but its knowledge is not so critical as warrant the use of a <tt>@@warning</tt> tag.<br>
250 + The Doxygen tag <tt>@@note</tt>, if it exists, must appear after all <tt>@@warning</tt> and <tt>@@attention</tt> and <tt>@@remark</tt> tags and
251 immediately before the ParaMonte custom command tag <tt>@@see</tt>if it exists, otherwise immediately before <tt>@@example</tt> for examples (if it exists).<br>
252 + The Doxygen tag <tt>@@see</tt>, if it exists, must appear after all <tt>@@warning</tt> and <tt>@@remark</tt> and <tt>@@note</tt> tags.<br>
253 If more than one item for the <tt>@@see</tt>command exists, each must be written on a separate line and each line must end with the HTML line-break tag `<br>`.
254 + If any example exists, it must appear immediately after the <tt>@@see</tt> tag,
255 otherwise after <tt>@@note</tt>, <tt>@@remark</tt>, <tt>@@warning</tt>, <tt>@@param</tt>, <tt>@@details</tt>, or <tt>@@brief</tt> if any exists.<br>
256 ParaMonte examples are initiated by the custom command <tt>@@example</tt> devised in the `config.txt` file of ParaMonte Doxygen documentation.<br>
257 If the example exists in an external file, then it must be included via the Doxygen <tt>@@include</tt> command,<br>
258 + immediately followed by the ParaMonte custom Doxygen command <tt>@@compile</tt> which inserts the generic example compile commands for the example.<br>
259 + optionally but immediately followed the output file of the example inserted in the documentation via the <tt>@@include</tt> command,<br>
260 + followed immediately by the inclusion of any other visualization or postprocessing scripts and output.<br>
261 <br>
262 <p></p>
263 <b>In all steps, it is imperative to not leave any empty lines between the successive commands of the example
264 section</b>, designated by the <tt>@@example</tt>, otherwise, each empty line will start a new paragraph in the documentation.<br>
265 + The Doxygen <tt>@@test</tt> tag, if any exists, must appear immediately after the example section designated by the <tt>@@example</tt> tag.<br>
266 + The Doxygen <tt>@@bug</tt> tag, if any exists, must appear immediately after the <tt>@@test</tt> tag or any other tag immediately preceding it.<br>
267 + The Doxygen <tt>@@todo</tt> tag, if any exists, must appear immediately after the <tt>@@todo</tt> tag or any other tag immediately preceding it.<br>
268 + The closing command of each documentation section must be the ParaMonte custom command <tt>@@final</tt> separated from the tags before and after by an empty line.<br>
269 + The Doxygen <tt>@@author</tt> tag is the last command to appear in any documentation section, and it must preferably have the format exemplified in the example below.<br>
270 <br>
271 <br>
272
273+ **ParaMonte Doxygen custom commands**.<br>
274 To simplify documentation and avoid retyping certain frequently used keywords and sentences, a number of Doxygen aliases are predfined
275 in the ParaMonte Doxygen `config.txt` file. These include (but are not limited to):
276 + <tt>@@warnpure</tt> Inserts a `@warning` about procedures that are `impure` when the library is built the preprocessor macro `CHECK_ENABLED=1`.
277 + <tt>@@elemental</tt> Inserts a <tt>@@remark</tt> tag indicating that the procedure of interest is `elemental`.
278 + <tt>@@pure</tt> Inserts a <tt>@@remark</tt> tag indicating that the procedure of interest is `pure`.
279 + <tt>@@interface</tt> Starts a **Possible calling interfaces** paragraph where different calling interfaces of a procedure can be listed.
280 + <tt>@@benchmark</tt> Starts a new **Benchmark** paragraph which is hyper-linked to the generic anchor `#benchmark` at the same location on the same page.
281 + <tt>@@benchmark{xxx}</tt> Starts a new **Benchmark** paragraph which is hyper-linked to the specific anchor `#benchmark-xxx` at the same location on the same page.
282 + <tt>@@benchmark{xxx, This is the benchmark title}</tt> Starts a new **Benchmark** paragraph which is hyper-linked to
283 the specific anchor `#benchmark-xxx` at the same location on the same page with the title `This is the benchmark title`.
284 + <tt>@@example</tt> Starts a new **Example usage** paragraph which is hyper-linked to the generic anchor `#example` at the same location on the same page.
285 + <tt>@@example{xxx}</tt> Starts a new **Example usage** paragraph which is hyper-linked to the specific anchor `#example-xxx` at the same location on the same page.
286 + <tt>@@compile</tt> Inserts the set of example compile commands.
287 + <tt>@@output</tt> Inserts a title line for the output section of an example paragraph.
288 + <tt>@@postproc</tt> Inserts a title line for the postprocessing section of an example paragraph.
289 + <tt>@@abbr</tt> Inserts a <tt>@@remark</tt> tag about the naming abbreviations used in the library.
290 + <tt>@@naming</tt> Inserts a <tt>@@remark</tt> tag about the naming conventions used in the library.
291 + <tt>@@license</tt> Inserts a <tt>@@remark</tt> tag about the generic licensing of the library.
292 + <tt>@@final</tt> Inserts the set of final generic remarks that should appear at the end of each documentation section.
293 <br>
294 <br>
295
296 For an up-to-date list of all available aliases, check the value of the Doxygen `ALIASES` option in `config.txt` in the ParaMonte Python documentation repository.
297
298+ **Escaping the Doxygen reserved characters**.<br>
299 Doxygen has a set of reserved characters whose usage in the documentation must be handled properly.<br>
300 + Most importantly, the backslash character `@` begins a Doxygen command.<br>
301 To print a backslash character to the output one should escape it via <tt>@@</tt>.<br>
302 + Also, the use of the percentage symbol `%` requires special care in some instances.<br>
303 This is particularly important when defining Windows environment variables that should typically be enclosed with percentage character.<br>
304 <br>
305
306 For more information, see [the relevant page on Doxygen documentation website](https://www.doxygen.nl/manual/commands.html#cmdfdollar).
307 <br>
308
309+ Avoid the insertion of an empty documentation line between any two lines of a single Doxygen paragraph.<br>
310 This is crucial when the whole paragraph is indented by a vertical line as is done by Doxygen for `@warning`, <tt>@@remark</tt>, `@note` and other similar tags.<br>
311 For an example see the documentation of [pm.sampling.Paradram](@ref Paradram) and the actual source code for it.<br>
