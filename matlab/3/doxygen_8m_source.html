9%> \section ParaMonte What is ParaMonte?
10%> <!--==============================-->
11%>
12%> ParaMonte is a library of serial and parallel Monte Carlo and Machine Learning routines
13%> scientific inference, e.g., for sampling mathematical density functions of arbitrary-dimensions,
14%> with the design goal of unifying
15%>
16%> + the **automation** of simulations and inference,
17%> + the **user-friendliness** of the library and routines,
18%> + the **accessibility** from multiple programming environments,
19%> + the **high-performance** at runtime, and,
20%> + the **scalability** across many parallel processors.

81%> \section ParaMonteLangNamingConventions ParaMonte MATLAB Naming Conventions
82%>
83%> + The **CamelCase** naming style is enforced throughout the ParaMonte MATLAB library.
84%>
85%> [⛓⛓](#ParaMonteLangNamingConventionsVariables)
86%> \subsection ParaMonteLangNamingConventionsVariables ParaMonte MATLAB Naming Conventions: Variables
87%>
88%> By convention in this library,
89%>
90%> + All variables and procedure names except compile-time constants begin with an lowercase letter.
91%>
92%> + All constants and parameters should generally be typed in uppercase.
93%>
94%> + The names of variables that **always** represent vectors of values **may** suffixed with `Vec` or `Vector`.
95%>
96%> + The names of variables that **always** represent matrices of values **may** be suffixed with `Mat` or `Matrix`.
97%>
98%> + A significant attempt has been made to end all Boolean variables with a passive verb.<br>
99%> This is to ensure that the full variable name virtually forms a proposition.<br>
100%> In other words, a Boolean variable name should be an English-language statement that evaluates to either `true` or `false`.<br>
101%> For example, `parallelismMpiFinalizeEnabled` is one such proposition.<br>
102%> + Occasionally, names that begin with the verb `is` can also be used to label `_Bool` objects.<br>
103%> + But as a general rule, names that begin with a verb should be reserved for procedures.<br>
104%>
105%> [⛓⛓](#ParaMonteLangNamingConventionsProcedures)
106%> \subsection ParaMonteLangNamingConventionsProcedures ParaMonte MATLAB Naming Conventions: Procedures
107%>
108%> + Procedure names should be descriptive of the action performed by the procedure. For example,
109%>
110%> + `getCov` means *generate (a) covariance matrix*.
111%> + `setMatCopy` means *copy the matrix into the specified buffer*.
112%>
113%> + Procedure names should virtually always begin with a lowercase verb.
114%>
115%> + **Exceptions** to the above and below rules are allowed when,
116%> + the procedure name is exceptionally famous, or
117%> + it is very inconvenient to prefix the procedure name with a verb.
118%>
119%> + The names of functions with return values should preferably begin with <b>`get`</b>.<br>
120%> The reasoning is simple: Functions in mathematics *generate and obtain* a **new** object instead of changing (resetting) the state of an existing object.<br>
121%> For example, the function `getMatSym(mat) result(matsym)` generates a symmetric version of the input matrix and returns it as the function result.<br>
122%> **Exceptions** to this naming convention are allowed, for example, when the `get` prefix is inconvenient or when the function returns a `_Bool` result.<br>
123%>
124%> + Functions that return objects of type `_Bool` should be preferably prefixed with `is` or
125%> be named such that the name begins with a verb and reads as a proposition, evaluating to
126%> either `.true.` or `.false.`.
127%>
128%> + The keyword <b>`get`</b> should be avoided as a prefix for `void` function names since `void` functions do not generate
129%> and <i>get</i> a new object as their results, rather they <b>(re)set</b> the state of existing objects passed to them.<br>
130%> + A more appropriate prefix for `void` functions is `set`, i.e., `setReplaced()`.
131%>

197%> \section ParaMonteLangDocumentationGuidelines ParaMonte MATLAB Documentation Guidelines
198%>
199%> + **Doxygen custom command orderings**.
200%>
201%> + The Doxygen tag `\brief` must always be the first line of the documentation of modules, types, and procedures.<br>
202%> Example: [pm_sampling](@ref pm_sampling).<br>
203%> + The Doxygen tag `\details`, if it exists, must always immediately follow the Doxygen tag `\brief`.<br>
204%> Example: [pm_sampling](@ref pm_sampling).<br>
205%> + The Doxygen tag `\param`, if any number of it exists, must always immediately follow the Doxygen tag `\brief` (or `\details` if it exists).<br>
206%> Example: [runParaDRAMD()](@ref pm_sampling::runParaDRAMD).<br>
207%> + The Doxygen tag `\return`, must be exclusively used to indicate the return value of functions.<br>
208%> If it exists, it must appear immediately after the set of `\param` tags. Example: [runParaDRAMD()](@ref pm_sampling::runParaDRAMD).<br>
209%> + If a generic interface is being documented, the ParaMonte custom command <tt>\\interface</tt> must appear immediately
210%> after the Doxygen `\return`, `\param`, `\details`, or `\brief` tags in the specified order, if any exists.<br>
211%> + The Doxygen tag `\warning`, if any number of it exists, must immediately follow the Doxygen tag `\return` if it exists,
212%> otherwise `\param` if it exists, otherwise `\details` if it exists, otherwise `\brief`.<br>
213%> The `\warning` tag must be used to highlight situations that require special attention of the user,
214%> otherwise, there is a danger for the code section being documented to not behave normally as one may expect.<br>
215%> + The Doxygen tag `\attention` has the same functionality and usage as `\warning`.<br>
216%> Therefore, `\warning` should be preferred wherever `\attention` is needed.<br>
217%> Exceptions are allowed and if they occur, the same documentation conventions as those of `\warning` also apply to the tag `\attention`.<br>
218%> + The Doxygen tag `\remark`, if any number of it exists, must immediately follow the Doxygen tag `\warning` if it exists,
219%> otherwise the Doxygen tag `\return` if it exists, otherwise `\param` if it exists, otherwise `\details` if it exists, otherwise `\brief`.<br>
220%> The tag `\remark` should be reserved for explaining behavior that is **directly** related to the code segment being documented,
221%> but its knowledge is not so critical as warrant the use of a `\warning` tag.<br>
222%> + The Doxygen tag `\note`, if it exists, must appear after all `\warning` and `\attention` and `\remark` tags and
223%> immediately before the ParaMonte custom command tag `\see` if it exists, otherwise immediately before <tt>\\example</tt> for examples (if it exists).<br>
224%> + The Doxygen tag `\see`, if it exists, must appear after all `\warning` and `\remark` and `\note` tags.<br>
225%> If more than one item for the `\see` command exists, each must be written on a separate line and each line must end with the HTML line-break tag `<br>`.
226%> Example: See [below](#example-ParaMonteLangDocumentationGuidelines).<br>
227%> + If any example exists, it must appear immediately after the `\see` tag, otherwise after `\note`, `\remark`, `\warning`, `\param`, `\details`, or `\brief` if any exists.<br>
228%> ParaMonte examples are initiated by the custom command <tt>\\example</tt> devised in the `config.txt` file of ParaMonte Doxygen documentation.<br>
229%> If the example exists in an external file, then it must be included via the Doxygen `\include` command, followed immediately by
230%> the ParaMonte custom Doxygen command <tt>\\compile</tt> which inserts the generic example compile commands for the example, followed optionally
231%> but immediately by the output file of the example inserted in the documentation via the `\include` command, followed immediately by the inclusion
232%> of any other visualization or postprocessing scripts and output.<br>
233%> <b>In all steps, it is imperative to not leave any empty lines between the successive commands of the example
234%> section</b>, designated by the <tt>\\example</tt>, otherwise, each empty line will start a new paragraph in the documentation.<br>
235%> Example: See [below](#example-ParaMonteLangDocumentationGuidelines).<br>
236%> + The Doxygen `\test` tag, if any exists, must appear immediately after the example section designated by the <tt>\\example</tt> tag.<br>
237%> + The Doxygen `\todo` tag, if any exists, must appear immediately after the `\test` tag or any other tag immediately preceding it.<br>
238%> + The Doxygen `\bug` tag, if any exists, must appear immediately after the `\todo` tag or any other tag immediately preceding it.<br>
239%> + The closing command of each documentation section must be the ParaMonte custom command <tt>\\finmain</tt> separated from the tags before and after by an empty line.<br>
240%> + The Doxygen `\author` tag is the last command to appear in any documentation section, and it must preferably have the format exemplified in the example below.<br>
241%> <br>
242%> <br>
243%>
244%> + **ParaMonte Doxygen custom commands**.<br>
245%> To simplify documentation and avoid retyping certain frequently used keywords and sentences, a number of Doxygen aliases are predfined
246%> in the ParaMonte Doxygen `config.txt` file. These include (but are not limited to):

251%> + <tt>\\benchmark</tt> Starts a new **Benchmark** paragraph which is hyper-linked to the generic anchor `#benchmark` at the same location on the same page.
252%> + <tt>\\benchmark{xxx}</tt> Starts a new **Benchmark** paragraph which is hyper-linked to the specific anchor `#benchmark-xxx` at the same location on the same page.
253%> + <tt>\\benchmark{xxx, This is the benchmark title}</tt> Starts a new **Benchmark** paragraph which is hyper-linked to
254%> the specific anchor `#benchmark-xxx` at the same location on the same page with the title `This is the benchmark title`.
255%> + <tt>\\example</tt> Starts a new **Example usage** paragraph which is hyper-linked to the generic anchor `#example` at the same location on the same page.

299%> \example{ParaMonteLangDocumentationGuidelines}
300%> The following is an example documentation for a procedure:
301%> \verbatim
302%>
303%> \ingroup pm_sampling
304%>
305%> \defgroup runParaDRAM runParaDRAM
306%>
307%> \brief
308%> Generate and return a non-zero value (`1`) if the procedure fails to fully accomplish the task of generating
309%> a Monte Carlo sample of the specified input mathematical objective function, otherwise, return `0`.
310%>
311%> \details
312%> This interface group is the entry point to all **MATLAB-style interfaces** to the ParaDRAM samplers of mathematical density functions.<br>
313%> Although the procedures of this generic interface return a single scalar of type `int32_t`, the procedures generate
314%> massive amounts of information about each simulation which are stored in appropriate external hard drive files.<br>
315%>
316%> See,
317%> <ol>
318%> <li> [this generic documentation page](\pmdoc_usage_sampling/paradram/output/)
319%> for more information on the generated output files for samplings performed using the [ParaDRAM](@ref pm_sampling::paradram_type) sampler.
320%> </ol>
321%>
322%> \param[in] getLogFunc : The input user-specified procedure pointer to the natural logarithm of the target density function.<br>
323%> On input, the function must take an input vector `state` of size `ndim` of type floating-point of kind \C_RKALL
324%> representing a state (point) from within the domain of the user-specified target density function whose function value must be returned.<br>
325%> On output, the user-specified procedure `getLogFunc()` must return the function value corresponding to the input `state[ndim]`.<br>
326%> The following illustrate the generic interface of input function pointer `getLogFunc(state, ndim)`,
327%> \code{.F90}
328%> REAL getLogFunc(REAL[] state, int32_t ndim);
329%> \endcode
330%> where `REAL` can be a floating-point type of kind \C_RKALL for the corresponding varying-precision sampler interfaces:<br>
331%> <ol>
332%> <li> [runParaDRAMF](@ref runParaDRAMF),
333%> <li> [runParaDRAMD](@ref runParaDRAMD),
334%> <li> [runParaDRAML](@ref runParaDRAML).
335%> </ol>
336%> \param[in] ndim : The input scalar constant of type `int32_t` representing the number of dimensions of the domain of the objective function.
337%> \param[in] input : The input scalar pointer of type `char` representing the null-terminated MATLAB string
338%> that contains either,
339%> <ol>
340%> <li> the path to an external input file containing the namelist group of ParaDRAM sampler specifications
341%> as outlined in the corresponding page of [ParaMonte library generic documentation website](\pmdoc_usage_sampling/paradram/specifications/).<br>
342%> <li> the namelist group of ParaDRAM sampler specifications as the can appear in an external input specification file.<br>
343%> </ol>
344%> While all input simulation specifications are optional, it is highly recommended to pay
345%> attention to the default settings of the domain boundaries and sampler starting point.<br>
346%> (**optional**. It is considered as missing if set to `NULL`.)
347%>
348%> \return
